# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Muffin Agent is a hierarchical multi-agent stock analysis system built with LangGraph. It orchestrates specialized agents that analyze stocks from multiple perspectives (technical, fundamental, news sentiment, etc.) to produce investment theses with price targets.

## Commands

```bash
# Install
pip install -e .          # runtime deps
pip install -e ".[dev]"   # + dev deps (ruff, mypy, pytest)

# Tests
pytest                           # all tests
pytest -m unit                   # unit tests only
pytest -m live                   # tests hitting live APIs
pytest tests/test_config.py      # single file
pytest --cov=muffin_agent tests/ # with coverage

# Code quality

mypy src/                        # type check
```

## Architecture

The source lives in `src/muffin_agent/` and is organized as:

- **`utils/base_config.py`** — `BaseConfiguration` Pydantic model. Shared base for all configuration classes. Provides `from_runnable_config(config: RunnableConfig)` classmethod which merges env vars with LangGraph's `RunnableConfig["configurable"]`.

- **`model_config.py`** — `ModelConfiguration(BaseConfiguration)`. Manages LLM provider selection (OpenAI/Anthropic/OpenRouter), API keys, and model parameters. Two construction surfaces:
  - `get_llm(model=None, temperature=None, **kwargs)` — single-model legacy path. `llm_provider: Literal["openai", "anthropic", "openrouter"]` picks the underlying class (`ChatOpenAI`, `ChatAnthropic`, `ChatOpenRouter`). `llm_sdk_retries` (default 6, env `LLM_SDK_RETRIES`) is forwarded to the SDK as `max_retries=` — covers connect-time errors before the response body arrives.
  - `get_llm_for_role(role)` — role-aware path. `role: Literal["orchestrator", "collector", "reasoner"]` picks the matching `*_models: list[str]` chain (env `ORCHESTRATOR_MODELS` / `COLLECTOR_MODELS` / `REASONER_MODELS` are comma-separated). Returns `list[BaseChatModel]` — first entry is the primary, the rest are fallbacks; always non-empty (when the role chain is unset, returns `[get_llm()]`). Per-entry instantiation reuses `get_llm(model=entry)`, so chain entries share `llm_provider`. Caller pattern: `primary, *fallbacks = config.get_llm_for_role("orchestrator")` then `MuffinAgentBuilder(primary).with_fallback_models(*fallbacks)`.
  - `get_summariser()` — optional cheap-and-fast model used by `ToolKnowledgeMiddleware` to LLM-summarise tool failures into one-line lessons. Returns a chat model built from `summariser_model` (env `SUMMARISER_MODEL`) or `None` when unset. Pair with `MuffinAgentBuilder.with_tool_knowledge(summariser)`; when `None`, skip the call — the middleware falls back to deterministic fallback lesson strings.
  - `DEFAULT_MODEL = "openai/gpt-oss-120b"` — note the missing `:free` suffix; free OpenRouter routes have aggressive rate limits and known mid-stream chunk-merge bugs and must be explicitly opted into per-deployment. Mid-stream and other transient errors are retried by LangChain's upstream `ModelRetryMiddleware`, wired by `MuffinAgentBuilder` (see below).

- **`mcp_config.py`** — `McpConfiguration(BaseConfiguration)`. Manages MCP server URLs: `openbb_mcp_url` (default `http://127.0.0.1:8001/mcp`) and `firecrawl_mcp_url` (default `http://127.0.0.1:3000/mcp`). `get_mcp_connections()` returns both; `MultiServerMCPClient` loads all tools and `get_tools()` filters by `allowed_tools`. Docker-compose sets `OPENBB_MCP_URL` and `FIRECRAWL_MCP_URL`. Used internally by `agents/data_collection/utils.py` and `middlewares/tool_result_cache/tools.py`.


- **`prompts/`** — Jinja2-based prompt templates organized into `data_collection/` and `investment/` subdirectories mirroring the `agents/` structure. Use `render_template(template_name, **kwargs)` from `prompts/__init__.py` with subdirectory-prefixed names (e.g., `render_template("data_collection/equity_fundamentals.jinja")`). Root-level agents (`stock_evaluation`, `criterion_evaluation`, `data_validation`) keep their templates at the prompts root. Investment prompts use shared Jinja2 partials (`investment/_data_rules.jinja`, `_validation_step.jinja`, `_returning_analysis.jinja`, `_missing_data_rules.jinja`) via `{% include %}` to avoid boilerplate duplication. Cross-cutting capability partials live under `middlewares/`: `short_term_memory.jinja` (`/scratch/` conventions), `persistent_memory.jinja` (`/memories/AGENTS.md` conventions), `skills.jinja` (read-only `/skills/` tree), `tool_result_cache.jinja` (cache workflow: discover → schema → write → load — appended by `with_sandbox()` only), `filesystem.jinja` (offload-recovery rule: when a tool result begins with "Tool result too large", call `read_file` on the provided path — appended by `build_react_agent()` universally for all ReAct agents), plus `sandbox.jinja` (execute_python + store↔sandbox bridge tools) and `tools/store.jinja` (generic CRUD tools). These partials are no longer `{% include %}`d by hand — `MuffinAgentBuilder` appends the matching partial whenever its `with_*` method is called, so the prompt author only writes the agent-specific body.

- **`utils/observability.py`** — Optional LangFuse tracing via `setup_tracing()`. Returns callback handlers for `RunnableConfig["callbacks"]`. Gracefully degrades if LangFuse is unavailable.

- **`utils/backends.py`** — `make_agent_backend(skills_root=_SKILLS_ROOT)`: convenience factory returning a `CompositeBackend` factory with four routes: default (`OpenSandboxBackend` via `sandbox.get_backend`), `/skills/` (read-only `FilesystemBackend`), `/scratch/` (`StateBackend`, thread-scoped), and `/memories/` (`StoreBackend` with per-user namespace `("memories", user_id)`). `_memories_namespace` resolves the namespace in three steps: (1) real `configurable.user_id`, (2) `MemoryConfiguration.memory_debug_user_id` (env `MEMORY_DEBUG_USER_ID` or `configurable.memory_debug_user_id`) as a debug-only fallback for local development against clients that don't populate `user_id` (e.g. agent-chat-ui), (3) raise `MemoryUnavailableError` (subclass of `LookupError`). The error propagates to the caller — set `MEMORY_DEBUG_USER_ID` in the dev stack to avoid it. `get_agent_backend = make_agent_backend()` is kept as a thin convenience for tests and ad-hoc callers; production agents compose their backends lazily via `MuffinAgentBuilder`.

- **`utils/memory_config.py`** — `MemoryConfiguration(BaseConfiguration)`. Single field `memory_debug_user_id: str | None` (env `MEMORY_DEBUG_USER_ID` or `configurable.memory_debug_user_id`). Consulted by `_memories_namespace` as a debug-only fallback when real `user_id` is missing. Do NOT set in multi-user deployments — collapses all anonymous traffic onto a single shared namespace.

- **`utils/agent_builder.py`** — `MuffinAgentBuilder(model, *, name=None)`: fluent builder for both deep agents (terminal `.build_deep_agent()`) and ReAct agents (terminal `.build_react_agent()`). Fully typed — no `Any` on the public surface. State is grouped into module-private dataclasses (`_PromptConfig`, `_BackendConfig`, `_ToolConfig`, `_SkillConfig`) so each `with_*` method touches one group and the terminal method is a single declarative `create_deep_agent(...)` / `create_agent(...)` call. Each `with_*` call enables one capability and registers: (a) the matching backend route/default maker, (b) the matching system-prompt partial (auto-appended by the terminal builder, except for capabilities whose middleware owns its own prompt content like `SubagentRefinement*Middleware`), (c) any required middleware.

  **Reliability / control methods** (added in the W1–W9 reliability waves):
  - `with_fallback_models(*models)` — registers `ModelFallbackMiddleware` outermost; each fallback is tried after retries on the current model are exhausted.
  - `with_context_editing(*, trigger=40_000, keep=4, clear_tool_inputs=False)` — wires upstream `ContextEditingMiddleware(ClearToolUsesEdit(...))` to trim older tool messages once the conversation crosses the trigger.
  - `with_summarization(model=None, *, trigger=("tokens", 80_000), keep=("messages", 20))` — wires upstream `SummarizationMiddleware`; `model` defaults to the agent's primary.
  - `with_model_call_limit(*, run_limit, thread_limit, exit_behavior="end")` — caps total LLM calls per run/thread (graceful end by default).
  - `with_tool_call_limit(*, tool_name=None, run_limit, thread_limit, exit_behavior="continue")` — global or per-tool tool-call cap. Multiple calls accumulate so per-tool and global caps coexist. Note: `with_tool()` already creates a per-tool `ToolCallLimitMiddleware` automatically (default `run_limit=6`) — call `with_tool_call_limit` only for global caps or when an explicit override is needed.
  - `with_tool_knowledge(summariser)` — upgrades `ToolKnowledgeMiddleware`'s deterministic fallback lessons to LLM-summarised ones (call once with a small/cheap model).
  - `with_subagent_refinement()` — opts into the structured-findings + scratch-cache refinement protocol. Role decided at build time: parent (subagents wired) → `SubagentRefinementParentMiddleware` (prompt-only); child (no subagents) → `SubagentRefinementMiddleware` (full child behaviour: forces `response_format=AutoStrategy(CollectionFindings)` if none set, reads/writes `/scratch/subagent_runs/<call_id>.json`, amends the system prompt with the static rules + any prior-call findings).

  **Composition primitives**: `with_system_prompt(str | SystemMessage)` / `with_system_prompt_template(name, **vars)` (mutually exclusive, last wins); `with_sandbox()` → default backend + registers all three cache tools (`discover_cached_tool_outputs`, `get_tool_output_schema`, `write_cached_tool_output_to_backend`) + appends `tool_result_cache.jinja` prompt; `with_short_term_memory(path="/scratch/")`; `with_persistent_memory(path="/memories/", filename="AGENTS.md")`; `with_skills(paths, *, skills_root=None, filter_middleware=None)` (deep-only, one bundle per agent); `with_subagents(Sequence[SubAgent | CompiledSubAgent])` (deep-only); `with_tool(tool, *, is_cacheable=True, run_limit=_DEFAULT_TOOL_RUN_LIMIT (6), thread_limit=None, exit_behavior="continue")` — per-tool policy declared inline; `run_limit` defaults to 6 (constant `_DEFAULT_TOOL_RUN_LIMIT`) and automatically creates a per-tool `ToolCallLimitMiddleware`; pass `run_limit=None` to opt out of the per-tool cap; `with_permission(FilesystemPermission)` (deep-only); `with_middleware(AgentMiddleware)`; `with_response_format`, `with_store(BaseStore)`, `with_context_schema(type)`, `with_checkpointer(Checkpointer)`.

  **Universal middleware order** (set by `_assemble_middleware`, outermost first): `ModelCallLimitMiddleware` (cond) → `ToolCallLimitMiddleware`s (cond, may repeat) → `ModelFallbackMiddleware` (cond) → `ModelRetryMiddleware` (universal, transient-only filter) → `ContextEditingMiddleware` (cond) → `SummarizationMiddleware` (cond) → `ToolKnowledgeMiddleware` (universal) → `ToolResultCacheMiddleware` (universal) → `ToolRetryMiddleware` (universal, 5xx/network filter) → ReAct-only `FilesystemMiddleware` / `MemoryMiddleware` → `SubagentRefinementMiddleware` or `SubagentRefinementParentMiddleware` (cond) → optional skill-filter middleware → caller-supplied via `with_middleware`. Backend routes for `StateBackend()` / `StoreBackend(namespace=...)` / `FilesystemBackend(...)` are instantiated eagerly on capability registration; only the sandbox default (`get_backend`) runs as a `BackendFactory` closure.

- **`middlewares/`** — Middleware package. Each subpackage is a focused concern:
  - `store_access/` — `StoreAccessMiddleware` plus `AccessControlledStore` and 5 generic store CRUD `@tool` functions (`store_get` / `store_put` / `store_delete` / `store_search` / `store_list_namespaces`). `StoreConfiguration.store_allowed_namespaces: list[str] | None` restricts which namespace prefixes an agent can access (`None` = unrestricted).
  - `tool_result_cache/middleware.py` — `ToolResultCacheMiddleware`: caches successful tool results in a shared `InMemoryStore` (via `ToolRuntime.store`) for cross-agent deduplication. **Strict-content invariant**: never mutates `ToolMessage.content` — tools that return JSON keep returning JSON. Cache provenance lives in `additional_kwargs["cache"]`. Caches both `str` and `list` content (the two valid `ToolMessage.content` types per LangChain) — this means Firecrawl tools (which return `list`) are cached and deduplicated. `_should_skip_cache(result, non_cacheable_patterns)` helper centralises skip logic: skips on `status='error'`, `is_error_content()` match (for `str` content), or any pattern in `non_cacheable_patterns` (for `str` content); `list` content with no error status is always cached. Size-based offloading of oversized payloads is **not** this middleware's job — that's `deepagents.middleware.filesystem.FilesystemMiddleware._aintercept_large_tool_result` (default 20K-token threshold). `cacheable_tools` parameter restricts which tools are cached (None = all). `self.tools = []` — cache tools are NOT registered here; they require sandbox and are registered by `with_sandbox()`. Also exports `get_args_hash()` and `is_error_content()` helpers.
  - `tool_result_cache/config.py` — `ToolResultCacheConfiguration(BaseConfiguration)`. `tool_schema_packages: list[str]` (default `["muffin_agent.tools", "muffin_agent.middlewares.store_access"]`) — decouples tool schema package scanning. `non_cacheable_patterns: list[str]` (default `["tool result too large"]`, env `TOOL_RESULT_CACHE_NON_CACHEABLE_PATTERNS`) — case-insensitive substrings matched against string content; results containing any pattern are excluded from cache. Default excludes `FilesystemMiddleware` offload messages which embed ephemeral `tool_call_id`-keyed paths that become stale on cache hit.
  - `tool_result_cache/tools.py` — `discover_cached_tool_outputs`, `get_tool_output_schema`, `write_cached_tool_output_to_backend`. **Registered by `with_sandbox()` only** — not by `ToolResultCacheMiddleware.tools` (which is empty). Non-sandbox agents benefit from automatic transparent cache hits in `awrap_tool_call` without needing the explicit cache tools in their prompt.
  - `tool_knowledge/` (replaces the older `tool_error_handler/`) — adaptive learning from tool failures. Split into focused modules:
    - `errors.py` — `is_permanent_error(msg)` and `duplicate_key(tool_call)` (pure logic).
    - `lessons.py` — `Lesson` dataclass + `LessonCatalog` wrapping a `BaseStore`; CRUD under namespace `("tool_lessons", tool_name)` keyed by `error_class_hash`.
    - `summariser.py` — `summarise_tool_failure(...)` (LLM-distilled lesson) with `fallback_lesson(tool, error)` (deterministic) for the no-summariser path.
    - `prompt.py` — `render_lessons_block(lessons)` and `append_block(existing, block)` for the system-prompt addendum.
    - `middleware.py` — `ToolKnowledgeMiddleware`: catches tool exceptions and errored `ToolMessage`s, blocks duplicate permanent failures (carried via `failed_tool_calls` state), records a lesson per unique `(tool, error_class)` pair (LLM-summarised when `with_tool_knowledge(summariser)` was called, deterministic fallback otherwise). **Extended permanent-error detection**: beyond `status='error'` and raised exceptions, also catches `ToolMessage`s with `status=None` whose string content matches `is_permanent_error()` (e.g. "OpenAI API key is missing…") — wraps the error as `"permanently unavailable in this environment: <content>"` so the recorded lesson is agent-facing (agent cannot configure API keys; lesson tells it not to retry). **Loop-pattern detection** (`loop_warning_threshold=3`): `awrap_model_call` counts tool calls per name across message history via `_count_tool_calls_in_history`; when any tool reaches the threshold, `_record_loop_lesson` is called with actual tool output samples (via `_get_recent_tool_outputs`) so the summariser can reason about why the tool isn't making progress (e.g. auth-gated content). Loop lessons use the same `LessonCatalog` + `summarise_tool_failure` pipeline as error lessons and flow into the same `## Lessons learned` block. `awrap_model_call` records loop lessons BEFORE reading the catalog so just-persisted lessons appear in the same call's block.
  - `subagent_refinement/` — generic conversational refinement protocol so subagents can be re-invoked to fill gaps without restarting:
    - `schema.py` — `CollectionFindings`, `Gap`, `GapReason` (StrEnum), `ToolCallSummary`. Generic `obtained: dict[str, Any]` so each subagent picks its own field names.
    - `storage.py` — `call_id_path`, `extract_prior_call_id` (regex with word-boundary lookahead), `read_findings`, `write_findings`, `latest_human_text` helpers.
    - `prompts.py` — thin facade over `render_template` that loads the role-specific instructions from the dedicated template directory (see "Prompt templates" note below) and renders the per-call prior-findings block.
    - `middleware.py` — two classes:
      - `SubagentRefinementMiddleware` (child) — `abefore_agent` discovers `prior_call_id=<id>` markers and stashes the prior findings on `prior_findings` state, `awrap_model_call` amends the system prompt with the static rules + (when present) the per-call prior block, `aafter_agent` writes this run's `CollectionFindings` to `/scratch/subagent_runs/<call_id>.json`. Auto-assigns `call_id` if missing.
      - `SubagentRefinementParentMiddleware` (parent) — only `awrap_model_call`; prepends the parent-side rules so the orchestrator knows how to read `gaps` and re-issue refinement calls with `prior_call_id=<id>`. No state, no tools.
  - `skill_suggestion/middleware.py` — `SkillFilterMiddleware(AgentMiddleware)`: schema-driven skill filtering middleware. Works alongside the default `SkillsMiddleware` (added via `skills=` on `create_deep_agent`). No tools registered. Parameterised via `__class_getitem__` with an `AgentState` subclass whose extra fields (beyond `AgentState`) become category keys: `SkillFilterMiddleware[TickerClassification](context_header=..., context_intro=..., context_outro=...)`. The three `context_*` constructor kwargs control the domain-specific text injected into the system prompt (defaults are domain-agnostic — "Classification" / "The current context has been classified as follows:" / standard outro). Classification is read from flat state keys (not a nested dict), so multiple agents can compose without key conflicts. `_filter_skills()` keeps skills whose ALL metadata category values match the classification; universal skills (no metadata) always match. `abefore_agent` filters `skills_metadata`, `awrap_model_call` injects classification context into the system prompt. Used by `criteria_definition.py` (ticker classification) and `agents/research/` (mode + task_type filtering).

**Universal model-retry layer.** `MuffinAgentBuilder._assemble_middleware` wires LangChain's upstream `ModelRetryMiddleware` to retry transient/mid-stream LLM provider errors that escape the SDK's connect-time `max_retries` (e.g. `openai.APIError("Provider returned error")` raised from inside the SSE stream after a 200 OK). Configured with `max_retries=3`, exponential backoff (`backoff_factor=2.0`, `initial_delay=1.0`, `max_delay=30.0`, `jitter=True`), and a `retry_on=` callable (`_should_retry_llm_call` in `agent_builder.py`) that explicitly excludes permanent errors (`AuthenticationError`, `PermissionDeniedError`, `BadRequestError` for both `openai` and `anthropic`) before matching transient ones. `on_failure="error"` re-raises after exhaustion so an outer `ModelFallbackMiddleware` can switch models. Knobs are hardcoded — tune by appending a custom `ModelRetryMiddleware` via `with_middleware(...)` if needed.

**Universal tool-retry layer.** Same file wires `ToolRetryMiddleware` (`max_retries=1`, `on_failure="continue"`, exponential backoff) with a `_should_retry_tool_call` filter that targets `ToolException` messages containing 5xx / gateway / connection / timeout substrings (plus bare `TimeoutError` / `ConnectionError`). 4xx, validation, and missing-credential errors propagate so the LLM (and `ToolKnowledgeMiddleware`) can adapt instead of burning retry budget. Sits below `ToolResultCacheMiddleware` so cache hits short-circuit and don't consume retry budget.

**Prompt templates owned by middlewares.** Capability prompt fragments live under `prompts/middlewares/`. Most are static partials auto-appended by `MuffinAgentBuilder` via `_PromptConfig.add_partial` (e.g. `tool_result_cache.jinja`, `short_term_memory.jinja`, etc.). The `subagent_refinement` capability owns its prompt content fully — templates live in `prompts/middlewares/subagent_refinement/{child_instructions,parent_instructions,prior_findings_block}.jinja`, loaded by `subagent_refinement/prompts.py` via `render_template` and amended onto `system_message` from inside `awrap_model_call`. The agent builder is **not** responsible for prompt construction for capabilities whose middleware owns its own prompt.

- **`agents/data_collection/`** — Data collection agents using MCP tools with the ReAct pattern. Each agent is a single `.py` file containing:
  - `MCP_TOOLS` — list of allowed MCP tool name strings
  - `create_*_agent(config: RunnableConfig)` — async; calls shared `get_tools()` for MCP tools, then composes the agent via `MuffinAgentBuilder(...).with_system_prompt_template(...).with_short_term_memory().with_tool(...).build_react_agent()`. Tools are cacheable by default; pass `is_cacheable=False` to opt out (e.g. `execute_python`).

  Shared utilities live in `utils.py`:
  - `get_tools(config: RunnableConfig, allowed_tools, custom_tools=None)` — calls `McpConfiguration.from_runnable_config(config)` to get all MCP connections (OpenBB + Firecrawl), loads all tools via `MultiServerMCPClient`, then filters by `allowed_tools`. Skips MCP entirely when `allowed_tools=[]`.

  Currently implemented: `equity_fundamentals.py` (25 OpenBB tools), `equity_price.py` (5 OpenBB tools + `execute_python`), `web_search.py` (6 Firecrawl MCP tools; SearxNG accessed via Firecrawl's `SEARXNG_ENDPOINT`)

- **`agents/data_validation.py`** — Pure reasoning agent (no MCP tools) that validates collected data against a criterion. Built with `create_agent(model, system_prompt=...)` and no tools — the ReAct loop resolves to a direct LLM response. Checks sufficiency, relevance, temporal validity, and consistency. Prompt: `data_validation.jinja`. Used as a `CompiledSubAgent` in both stock evaluation and criterion evaluation agents.

- **`agents/subagents.py`** — Shared subagent builders. `build_analysis_subagents(config)` creates the standard 15 subagents (14 data collection + 1 validation) used by `stock_evaluation.py` and `criterion_evaluation.py`. `build_validation_subagent(config)` creates just the data-validation subagent, used by all 4 investment stage agents.

- **`agents/investment/schemas.py`** — Shared Pydantic models used across investment stage agents (`DataSource`).

- **`agents/investment/utils.py`** — Shared `run_deep_agent_node()` utility that encapsulates the pattern common to all investment node functions: `ModelConfiguration` extraction → agent creation (with optional `store` param) → state context → ainvoke → structured output extraction → error fallback with exception handling. Accepts `store: BaseStore | None` and passes it to agent factories as `store=store` kwarg.

- **`agents/criterion_evaluation.py`** — Deep agent that evaluates a single investment criterion. Uses `build_analysis_subagents()` and `create_deep_agent()`. Follows a 5-step workflow: Analyze Criterion → Collect Data → Validate → Evaluate (CoT with dynamic sub-criteria) → Reflect. Output enforced via `AutoStrategy(schema=CriterionEvaluationOutput)` — co-located Pydantic model exposing `criterion_name`, `score`, `confidence`, `signal` (5-level Literal), `sub_criteria`, `evidence_summary`, `reasoning`, `counterargument`, `limitations`, `data_sources`. Consumed structurally by the criteria-driven orchestrator.

- **`agents/criteria_analysis/`** — Criteria-driven analysis orchestrator graph. Top-level pipeline that:
  1. **Stage 1 — `ticker_classification_node`**: lightweight deep agent (3 data subagents: etf-index, equity-fundamentals, economy-macro + data-validation) producing `TickerClassificationOutput` (sector / sub_sector / market / stock_type). Short-circuits when the caller pre-supplies all four flat state keys (CLI flags). Lifts classification into the flat state keys read by `SkillFilterMiddleware[TickerClassification]` downstream.
  2. **Stage 2 — `criteria_definition_node`** (parallel with Stage 3): thin wrapper around `create_criteria_definition_agent` reading the flat classification keys to filter `/skills/valuation/`.
  3. **Stage 3 — `valuation_methodology_node`** (parallel with Stage 2): deep agent with `web-search` + `discovery-screening` subagents producing `ValuationMethodologyOutput` (methodology summary + 2–5 ticker-specific additional criteria). Deliberately complementary to the skill-filtered stream — surfaces narrative/governance/activism/regulatory criteria the sector skills miss.
  4. **Stage 4a — `merge_criteria_node`**: pure-Python deterministic dedup. Lowercases + strips punctuation → canonical name; skill version wins on tie; re-normalises weights to sum to 1.0; tags each survivor with `source: "skill" | "web"`.
  5. **Stage 4b — `criterion_evaluation_node`** (fan-out): one `Send` per merged criterion forwarding ticker + query + classification + criterion. Each worker invokes the structured `criterion_evaluation` agent and returns `{"criterion_evaluations": [evaluation]}` with the criterion's `weight` + `source` carried through. `operator.add` reducer accumulates per-criterion results across parallel workers.
  6. **Stage 5 — `synthesis_node`**: reasoning-only deep agent (no subagents, no tools) producing `CriteriaAnalysisSynthesis` — composite score (weight-weighted), buy/sell/hold signal, weighted breakdown, key positives/negatives, divergences, confidence (floor of classification + weighted criterion confidences), thesis paragraph.

  Implicit barriers (LangGraph fires a node only when all incoming edges have data) handle Stage 2/3 → Stage 4a synchronisation — same pattern as [investment_analysis.py:64-85](src/muffin_agent/agents/investment_analysis.py#L64-L85). Send fan-out + `operator.add` reducer mirrors [equity_screening.py:61-79](src/muffin_agent/agents/equity_screening.py#L61-L79). Per-criterion concurrency is uncapped.

  Public entry: `build_criteria_analysis_graph(checkpointer=None, store=None) -> CompiledStateGraph`. Module-level `graph` variable exposed at `src/muffin_agent/agents/criteria_analysis/graph.py:graph` for LangGraph Platform autodiscovery; also registered in [langgraph.json](langgraph.json) as `"criteria_analysis"`. Schemas live in `criteria_analysis/schemas.py` (`TickerClassificationOutput`, `ValuationMethodologyOutput`, `CriteriaAnalysisSynthesis`, `WeightedBreakdownEntry`). State in `criteria_analysis/state.py` (`CriteriaAnalysisState` with flat classification keys + `criterion_evaluations: Annotated[list, operator.add]`). Shared invocation helper `invoke_structured_agent` in `criteria_analysis/_node_helpers.py` mirrors `investment.utils.run_deep_agent_node` but passes the original `RunnableConfig` straight to factories (avoids the latent `ModelConfiguration` ↔ `RunnableConfig` confusion that affects the older helper). CLI: `muffin criteria-analyze AAPL [--query …] [--sector …] [--market …] [--stock-type …]`.

- **`agents/trading_decision/`** — Composable trading-decision pipeline ported from [TauricResearch/TradingAgents](https://github.com/TauricResearch/TradingAgents). Decoupled from `investment_analysis` — agents accept a generic `AnalysisContext` envelope so callers can feed in muffin's pipeline output, free-form research notes, or any other upstream analysis. Roll-out plan (5 PRs):
  - **PR 1 (shipped):** Bull / Bear Researcher debate + Investment Judge synthesis (`build_investment_debate_graph`).

- **`agents/research/`** — General-purpose deep research subgraph (domain-agnostic, Perplexity-style). Linear LangGraph pipeline: `classifier → researcher → rerank → writer`. Inspired by Vane (Perplexica) — single-classifier-with-flags shape, embedding-rerank step (cosine ≥ 0.5, top-K 20), inline `[N]` citation discipline. Nodes:
  1. **`classifier_node`** — plain ReAct agent (no tools, collector role) with `response_format=AutoStrategy(ResearchClassification)`. Output: `standalone_query` (coref-resolved), `task_type` (research_report/comparison/how_to/summary/debate/factual_qa), `mode_hint` (speed/balanced/quality), `sources_to_use`, `skip_search`. Lifted into flat state keys so downstream nodes — and the researcher's `SkillFilterMiddleware` — can read them directly. Routes to `writer` directly on `skip_search=True`.
  2. **`researcher_node`** — deep agent built via `MuffinAgentBuilder`. Tools: `firecrawl_search` + `firecrawl_scrape` from the existing Firecrawl MCP (via `get_tools(config, ["firecrawl_search", "firecrawl_scrape"])`) plus any caller-supplied `extra_tools`. Skills: `with_skills(["/skills/research/"], filter_middleware=SkillFilterMiddleware[ResearchClassificationFilterState](context_header="Research Configuration", ...))`. Mode-driven `with_model_call_limit` (speed=2 / balanced=6 / quality=25). `response_format=AutoStrategy(ResearchEvidenceFindings)`. Plus `with_short_term_memory()` (/scratch/) + `with_persistent_memory()` (/memories/). Emits `evidence_chunks` via the structured response — free-form messages are discarded.
  3. **`rerank_node`** — pure Python (no LLM). Embeds query + each evidence chunk via `langchain_openai.OpenAIEmbeddings` (configurable `base_url` / `api_key` — works with OpenAI direct, OpenRouter incl. free `nvidia/llama-nemotron-embed-vl-1b-v2:free`, vLLM, LM Studio, Ollama). Cosine-filters at `rerank_threshold` (default 0.5), URL-dedups with content-merge (keeps longer body), caps at `rerank_top_k` (default 20). Vane parity.
  4. **`writer_node`** — plain ReAct agent (no tools, orchestrator role) with `response_format=AutoStrategy(ResearchOutput)`. Receives reranked evidence rendered in the prompt with `[N]` markers; produces markdown answer with inline `[N]` citations, key findings, source list, confidence, missing information, suggested follow-ups. Task-type-specific output shape baked into the Jinja template.

  Public entries: `build_research_graph(checkpointer=None, store=None, extra_tools=None, extra_sources=None) -> CompiledStateGraph` (module-level `graph` registered in [langgraph.json](langgraph.json) as `"research"`) and `build_research_subagent(config, extra_tools=None, extra_sources=None, name="deep-research") -> CompiledSubAgent` for embedding inside other deep agents. Pluggability: callers pass `extra_tools` (additional `BaseTool` instances — e.g. academic / news / finance / internal-docs search) and `extra_sources` (source names to expose in the classifier's enum). Source gating is prompt-driven (Vane parity), not tool-list-filtered. Schemas in `agents/research/schemas.py`. State in `agents/research/state.py` (`ResearchState` with flat classification keys + `evidence: Annotated[list, operator.add]`; `ResearchClassificationFilterState` is the minimal schema fed to `SkillFilterMiddleware`). Config: `ResearchConfiguration(BaseConfiguration)` (env: `EMBEDDING_MODEL`, `EMBEDDING_BASE_URL`, `EMBEDDING_API_KEY`, `RERANK_THRESHOLD`, `RERANK_TOP_K`, `RESEARCH_DEFAULT_MODE`, `RESEARCH_DEFAULT_SOURCES`, `MAX_SEARCH_RESULTS`, `RESEARCH_ITER_SPEED/BALANCED/QUALITY`). Skills under `src/muffin_agent/skills/research/` (modes/{speed,balanced,quality}, task_types/{research_report,…}, plus universal `_shared/citation_discipline`). Subagent refinement protocol NOT used — research's contract is the complete `ResearchOutput`, not partial findings; follow-ups go via re-invocation with new query + checkpointer thread history. **Migration note**: the researcher is currently a single `create_deep_agent` call. If we need streaming progress to a UI panel (Vane-style block streaming) or observe poor LLM sequencing in LangSmith, swap `researcher_node` internals to a multi-node sub-graph (plan → search → scrape → decide_continue → consolidate). The state contract (`state["query"] → state["evidence"]`) is the only public surface — migration is non-breaking. CLI: `muffin research "QUERY" [--mode speed|balanced|quality] [--sources web,...] [--task-type ...] [--user ...] [--thread ...]`.

- **`sandbox/`** — OpenSandbox integration. Sandboxes are discovered/created lazily by `thread_id` metadata — no middleware or state propagation needed.
  - `OpenSandboxConfiguration` (`config.py`) — `BaseConfiguration` subclass with `opensandbox_url`, `opensandbox_api_key`, and `opensandbox_image` fields.
  - `OpenSandboxBackend` (`backend.py`) — `deepagents.BaseSandbox` implementation backed by a `SandboxSync` container. Pure wrapper; all file operations delegated to shell commands via `execute()`.
  - `SandboxFactory` (`factory.py`, internal) — Discovers running sandboxes by `thread_id` metadata via `SandboxManagerSync.list_sandbox_infos()`. Creates a new sandbox if none found. Works with both `ToolRuntime` and `Runtime` contexts (`thread_id` from `langgraph.config.get_config()`).

  - `get_sandbox` / `aget_sandbox` (`factory.py`) — Sync/async functions returning a raw sandbox instance for direct use.
  - `execute_python` (`tools.py`) — `@tool` async tool. Discovers the sandbox for the current thread via `aget_sandbox` and executes Python code in it. Used for ad-hoc calculations not covered by the financial tools in `tools/`.
  - `write_store_data_to_sandbox` (`tools.py`) — `@tool` async tool. Reads any store entry via `AccessControlledStore` and writes its value as JSON to the sandbox filesystem (default `/data/store/{namespace}/{key}.json`).
  - `read_sandbox_file_to_store` (`tools.py`) — `@tool` async tool. Reads a sandbox file and puts its content into the store via `AccessControlledStore`. Non-JSON content is wrapped in `{"content": ...}`.
  - Cache-related tools (`discover_cached_tool_outputs`, `get_tool_output_schema`, `write_cached_tool_output_to_backend`) live in `middlewares/tool_result_cache/tools.py`.

- **`tools/`** — LangChain `@tool(parse_docstring=True)` functions for deterministic financial computations, organized by domain. Each tool contains its own computation logic inline. Parameter descriptions are parsed from Google-style docstrings into the JSON schema so the LLM sees full context. Tools are passed to `create_deep_agent` via `tools=[...]` — each investment agent imports only its relevant subset. Runs in-process (no sandbox round-trip). Unit tested in `tests/tools/`. Tools that return structured JSON define a co-located Pydantic output model and attach its pre-computed schema via `@tool(extras={"output_schema": Model.model_json_schema()})`. The tool validates output at construction time via `Model(...).model_dump()`, and `get_tool_output_schema` auto-discovers the schema by scanning `extras`.

  - `credit_risk.py` — `compute_net_debt_to_ebitda`, `compute_interest_coverage`, `compute_altman_z_score` (used by company_analysis)
  - `sector.py` — `compute_sector_relative_performance`, `compute_peer_dispersion` (used by market_regime, sector_analysis)

## Memory

Filesystem routes are composed opt-in through `MuffinAgentBuilder`. Every capability the builder opts into comes with (a) a `CompositeBackend` route, (b) the matching prompt partial teaching the LLM how to use it, and (c) any required middleware. The four available routes:

- **`/memories/`** (long-term, per-user) — enabled via `.with_persistent_memory()`. Backed by `StoreBackend` namespaced to `("memories", user_id)`. Deep agents get the stock `MemoryMiddleware` implicitly via `create_deep_agent(memory=[...])`; ReAct agents get `MemoryMiddleware` wired explicitly in `_assemble_middleware`. Both auto-load `AGENTS.md` into the system prompt. `_memories_namespace` resolves the namespace in three steps: `configurable.user_id` → `MemoryConfiguration.memory_debug_user_id` (env `MEMORY_DEBUG_USER_ID` or `configurable.memory_debug_user_id`) as a debug-only fallback → raise `MemoryUnavailableError`. CLI: pass `--user <id>`. Server: set `configurable.user_id` per request (via LangGraph auth hook once multi-user is actually required). Dev stack: set `MEMORY_DEBUG_USER_ID` in `.env` or on the `langgraph-api` service — do NOT set in multi-user deployments.
- **`/scratch/`** (short-term, thread-scoped) — enabled via `.with_short_term_memory()`. Backed by `StateBackend`, checkpointed with graph state. Survives `execute_python` sandbox container recycling; cleared when the thread is deleted.
- **`/skills/`** (read-only) — enabled via `.with_skills(paths, filter_middleware=...)` (deep-agent only). Backed by `FilesystemBackend(virtual_mode=True)` rooted at the shipped `skills/` directory. Currently used by `criteria_definition.py` (`/skills/valuation/`) and `agents/research/` (`/skills/research/` — modes + task_types). Research evidence cache lives under `/scratch/research/<thread_id>/` (created on demand by the researcher).
- **Default (no prefix)** — enabled via `.with_sandbox()`. Routes to `OpenSandboxBackend` — where `execute_python` runs and where auto-offloaded tool outputs (>20k tokens) land. Ephemeral.

User isolation: the `user_id` key flows from `--user` (CLI) → `configurable={"user_id": ...}` → `_memories_namespace(rt)`. Each user gets an isolated namespace. Local dev against clients that don't populate `user_id` (e.g. `agent-chat-ui`) set `MEMORY_DEBUG_USER_ID` so the fallback pins them to a shared namespace; without it `_memories_namespace` raises `MemoryUnavailableError`. Multi-user deployments must populate `configurable.user_id` per request to get isolation and must NOT set `MEMORY_DEBUG_USER_ID` (it collapses all anonymous traffic onto a single shared namespace).

Data-collection ReAct sub-agents opt into `/scratch/` via `.with_short_term_memory()` and get `FilesystemMiddleware` wired automatically; they do **not** call `.with_persistent_memory()`.

## Conventions

- **Ruff** with Google-style docstrings (`D401` imperative mood enforced). `D` and `UP` rules are relaxed in `tests/`.
- **Pydantic** models for all configuration and structured outputs.
- Python 3.11+ required.
- Environment variables for secrets (API keys, LangFuse credentials). See `.env` file.
- MCP (Model Context Protocol) for external data access (OpenBB).
- Use `create_agent` from `langchain.agents` for ReAct agents — do NOT use `create_react_agent` from `langgraph.prebuilt` (deprecated).
- KISS: no empty files or premature abstractions. Extract shared utilities only when patterns emerge across multiple agents.

## Collaboration Preferences

These rules govern how Claude approaches planning, implementation, and communication in this project.

1. **Deep planning first** — Always do deep planning and trade-off evaluation before writing any code. Explore the solution space thoroughly before committing to an approach.

2. **Prefer out-of-the-box solutions** — Before implementing custom logic, research available library features by reading internet documentation and/or library source code. Consider alternative options even if they are not an exact match to the ask. Surface interesting options proactively.

3. **Propose options, don't decide** — When facing a design decision or when multiple approaches exist, present the options and ask for a decision rather than picking one unilaterally. Ask questions before writing substantial code if no existing library/utility has been found — the user may be able to provide documentation pointers.

4. **Explicit approval before implementation** — Always ask for explicit approval before starting implementation. Never exit plan mode unless the user explicitly says to exit or switch mode.

5. **Keep documentation up to date** — After every implementation, update README.md, docs/, roadmap.md, and any other relevant docs as applicable. Add VSCode launch configurations where reasonable. Always include documentation updates as the last step of implementation plans. When trade-offs or tech debt are accepted, document the limitations and add action items to roadmap.

6. **Memorize lessons in CLAUDE.md** — If the user shares information that will be useful in future sessions (e.g. future roadmap tasks, corrections, disagreements, repeating feedback patterns, new constraints), record it in CLAUDE.md. When in plan mode, include the CLAUDE.md memory update as an explicit plan step.
