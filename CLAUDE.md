# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Muffin Agent is a hierarchical multi-agent stock analysis system built with LangGraph. It orchestrates specialized agents that analyze stocks from multiple perspectives (technical, fundamental, news sentiment, etc.) to produce investment theses with price targets.

## Commands

```bash
# Install
pip install -e .          # runtime deps
pip install -e ".[dev]"   # + dev deps (ruff, mypy, pytest)

# Tests
pytest                           # all tests
pytest -m unit                   # unit tests only
pytest -m live                   # tests hitting live APIs
pytest tests/test_config.py      # single file
pytest --cov=muffin_agent tests/ # with coverage

# Code quality
ruff check src/ tests/           # lint
ruff format src/ tests/          # format
mypy src/                        # type check
```

## Architecture

The source lives in `src/muffin_agent/` and is organized as:

- **`utils/base_config.py`** ΓÇö `BaseConfiguration` Pydantic model. Shared base for all configuration classes. Provides `from_runnable_config(config: RunnableConfig)` classmethod which merges env vars with LangGraph's `RunnableConfig["configurable"]`.

- **`model_config.py`** ΓÇö `ModelConfiguration(BaseConfiguration)`. Manages LLM provider selection (OpenAI/Anthropic/OpenRouter), API keys, and model parameters. Two construction surfaces:
  - `get_llm(model=None, temperature=None, **kwargs)` ΓÇö single-model legacy path. `llm_provider: Literal["openai", "anthropic", "openrouter"]` picks the underlying class (`ChatOpenAI`, `ChatAnthropic`, `ChatOpenRouter`). `llm_sdk_retries` (default 6, env `LLM_SDK_RETRIES`) is forwarded to the SDK as `max_retries=` ΓÇö covers connect-time errors before the response body arrives.
  - `get_llm_for_role(role)` ΓÇö role-aware path. `role: Literal["orchestrator", "collector", "reasoner"]` picks the matching `*_models: list[str]` chain (env `ORCHESTRATOR_MODELS` / `COLLECTOR_MODELS` / `REASONER_MODELS` are comma-separated). Returns `list[BaseChatModel]` ΓÇö first entry is the primary, the rest are fallbacks; always non-empty (when the role chain is unset, returns `[get_llm()]`). Per-entry instantiation reuses `get_llm(model=entry)`, so chain entries share `llm_provider`. Caller pattern for agent factories that wire fallbacks into `MuffinAgentBuilder`: `primary, *fallbacks = config.get_llm_for_role("orchestrator")` then `MuffinAgentBuilder(primary).with_fallback_models(*fallbacks)`.
  - `get_chat_model_for_role(config, role, *, schema=None, stop_after_attempt=3)` ΓÇö classmethod convenience for graph nodes that call an LLM directly (Bull / Bear / Judge / Trader / risk debators / PM / reflection). Returns a composed `Runnable` ready to `ainvoke`: primary + fallbacks (via `with_fallbacks`) + LangChain `with_retry(stop_after_attempt=3, wait_exponential_jitter=True)`. Replaces the 5-line boilerplate (`ModelConfiguration.from_runnable_config(...).get_llm_for_role(...)` ΓåÆ `with_fallbacks` ΓåÆ `with_retry`) that nodes would otherwise repeat at every call site. **Pass `schema=Schema` for structured-output nodes** (Judge / Trader / PM) ΓÇö the helper applies `with_structured_output(schema)` to the primary AND each fallback BEFORE composing with fallbacks/retry. Overload-typed: returns `Runnable[..., AIMessage]` when `schema=None`, `Runnable[..., BaseModel | dict]` when `schema=Schema`. **Do NOT chain `.with_structured_output(...)` or `.bind_tools(...)` onto the helper's return** ΓÇö the runtime object is a `RunnableRetry` (not a `BaseChatModel`), and `RunnableRetry` does NOT proxy chat-model methods (the `__getattr__` magic lives on `RunnableBinding`, not on `RunnableBindingBase` which is `RunnableRetry`'s parent). Such chained calls raise `AttributeError` at runtime. The `schema=` parameter is the supported path for structured output; for `bind_tools` on a non-builder agent, bind on the primary + each fallback before they reach `get_llm_for_role`-style plumbing (not currently needed in this codebase). Regression test guarding the chain order: `tests/test_config.py::TestGetChatModelForRoleRegression`.
  - `get_summariser()` ΓÇö optional cheap-and-fast model used by `ToolKnowledgeMiddleware` to LLM-summarise tool failures into one-line lessons. Returns a chat model built from `summariser_model` (env `SUMMARISER_MODEL`) or `None` when unset. Pair with `MuffinAgentBuilder.with_tool_knowledge(summariser)`; when `None`, skip the call ΓÇö the middleware falls back to deterministic fallback lesson strings.
  - `DEFAULT_MODEL = "openai/gpt-oss-120b"` ΓÇö note the missing `:free` suffix; free OpenRouter routes have aggressive rate limits and known mid-stream chunk-merge bugs and must be explicitly opted into per-deployment. Mid-stream and other transient errors are retried by LangChain's upstream `ModelRetryMiddleware`, wired by `MuffinAgentBuilder` (see below).

- **`mcp_config.py`** ΓÇö `McpConfiguration(BaseConfiguration)`. Manages MCP server URLs: `openbb_mcp_url` (default `http://127.0.0.1:8001/mcp`) and `firecrawl_mcp_url` (default `http://127.0.0.1:3000/mcp`). `get_mcp_connections()` returns both; `MultiServerMCPClient` loads all tools and `get_tools()` filters by `allowed_tools`. Docker-compose sets `OPENBB_MCP_URL` and `FIRECRAWL_MCP_URL`. Used internally by `agents/data_collection/utils.py` and `middlewares/tool_result_cache/tools.py`.


- **`prompts/`** ΓÇö Jinja2-based prompt templates organized into `data_collection/` and `investment/` subdirectories mirroring the `agents/` structure. Use `render_template(template_name, **kwargs)` from `prompts/__init__.py` with subdirectory-prefixed names (e.g., `render_template("data_collection/equity_fundamentals.jinja")`). Root-level agents (`stock_evaluation`, `criterion_evaluation`, `data_validation`) keep their templates at the prompts root. Investment prompts use shared Jinja2 partials (`investment/_data_rules.jinja`, `_validation_step.jinja`, `_returning_analysis.jinja`, `_missing_data_rules.jinja`) via `{% include %}` to avoid boilerplate duplication. Cross-cutting capability partials live under `middlewares/`: `short_term_memory.jinja` (`/scratch/` conventions), `persistent_memory.jinja` (`/memories/AGENTS.md` conventions), `skills.jinja` (read-only `/skills/` tree), `tool_result_cache.jinja` (cache workflow: discover ΓåÆ schema ΓåÆ write ΓåÆ load ΓÇö appended by `with_sandbox()` only), `filesystem.jinja` (offload-recovery rule: when a tool result begins with "Tool result too large", call `read_file` on the provided path ΓÇö appended by `build_react_agent()` universally for all ReAct agents), plus `sandbox.jinja` (execute_python + storeΓåösandbox bridge tools) and `tools/store.jinja` (generic CRUD tools). These partials are no longer `{% include %}`d by hand ΓÇö `MuffinAgentBuilder` appends the matching partial whenever its `with_*` method is called, so the prompt author only writes the agent-specific body.

- **`utils/observability.py`** ΓÇö Optional LangFuse tracing via `setup_tracing()`. Returns callback handlers for `RunnableConfig["callbacks"]`. Gracefully degrades if LangFuse is unavailable.

- **`utils/backends.py`** ΓÇö `make_agent_backend(skills_root=_SKILLS_ROOT)`: convenience factory returning a `CompositeBackend` factory with four routes: default (`OpenSandboxBackend` via `sandbox.get_backend`), `/skills/` (read-only `FilesystemBackend`), `/scratch/` (`StateBackend`, thread-scoped), and `/memories/` (`StoreBackend` with per-user namespace `("memories", user_id)`). `_memories_namespace` resolves the namespace in three steps: (1) real `configurable.user_id`, (2) `MemoryConfiguration.memory_debug_user_id` (env `MEMORY_DEBUG_USER_ID` or `configurable.memory_debug_user_id`) as a debug-only fallback for local development against clients that don't populate `user_id` (e.g. agent-chat-ui), (3) raise `MemoryUnavailableError` (subclass of `LookupError`). The error propagates to the caller ΓÇö set `MEMORY_DEBUG_USER_ID` in the dev stack to avoid it. `get_agent_backend = make_agent_backend()` is kept as a thin convenience for tests and ad-hoc callers; production agents compose their backends lazily via `MuffinAgentBuilder`.

- **`utils/memory_config.py`** ΓÇö `MemoryConfiguration(BaseConfiguration)` plus the shared `MEMORIES_NAMESPACE_ROOT = ("memories",)` constant and the `MemoryUnavailableError(LookupError)` exception (the canonical home; `utils/backends.py` re-exports for backwards compatibility). Single field `memory_debug_user_id: str | None` (env `MEMORY_DEBUG_USER_ID` or `configurable.memory_debug_user_id`). Two classmethods: `from_runnable_config(config)` (inherited from `BaseConfiguration`) and `resolve_user_id(config, *, allow_missing=False)` ΓÇö the single source of truth for the `configurable.user_id` ΓåÆ `memory_debug_user_id` fallback chain used by `_memories_namespace` (raises) and by the trading_decision reflection bookends (passes `allow_missing=True` to degrade gracefully). Do NOT set `memory_debug_user_id` in multi-user deployments ΓÇö collapses all anonymous traffic onto a single shared namespace.

- **`utils/agent_builder.py`** ΓÇö `MuffinAgentBuilder(model, *, name=None)`: fluent builder for both deep agents (terminal `.build_deep_agent()`) and ReAct agents (terminal `.build_react_agent()`). Fully typed ΓÇö no `Any` on the public surface. State is grouped into module-private dataclasses (`_PromptConfig`, `_BackendConfig`, `_ToolConfig`, `_SkillConfig`) so each `with_*` method touches one group and the terminal method is a single declarative `create_deep_agent(...)` / `create_agent(...)` call. Each `with_*` call enables one capability and registers: (a) the matching backend route/default maker, (b) the matching system-prompt partial (auto-appended by the terminal builder, except for capabilities whose middleware owns its own prompt content like `SubagentRefinement*Middleware`), (c) any required middleware.

  **Reliability / control methods** (added in the W1ΓÇôW9 reliability waves):
  - `with_fallback_models(*models)` ΓÇö registers `ModelFallbackMiddleware` outermost; each fallback is tried after retries on the current model are exhausted.
  - `with_context_editing(*, trigger=40_000, keep=4, clear_tool_inputs=False)` ΓÇö wires upstream `ContextEditingMiddleware(ClearToolUsesEdit(...))` to trim older tool messages once the conversation crosses the trigger.
  - `with_summarization(model=None, *, trigger=("tokens", 80_000), keep=("messages", 20))` ΓÇö wires upstream `SummarizationMiddleware`; `model` defaults to the agent's primary.
  - `with_model_call_limit(*, run_limit, thread_limit, exit_behavior="end")` ΓÇö caps total LLM calls per run/thread (graceful end by default).
  - `with_tool_call_limit(*, tool_name=None, run_limit, thread_limit, exit_behavior="continue")` ΓÇö global or per-tool tool-call cap. Multiple calls accumulate so per-tool and global caps coexist. Note: `with_tool()` already creates a per-tool `ToolCallLimitMiddleware` automatically (default `run_limit=6`) ΓÇö call `with_tool_call_limit` only for global caps or when an explicit override is needed.
  - `with_tool_knowledge(summariser)` ΓÇö upgrades `ToolKnowledgeMiddleware`'s deterministic fallback lessons to LLM-summarised ones (call once with a small/cheap model).
  - `with_subagent_refinement()` ΓÇö opts into the structured-findings + scratch-cache refinement protocol. Role decided at build time: parent (subagents wired) ΓåÆ `SubagentRefinementParentMiddleware` (prompt-only); child (no subagents) ΓåÆ `SubagentRefinementMiddleware` (full child behaviour: forces `response_format=AutoStrategy(CollectionFindings)` if none set, reads/writes `/scratch/subagent_runs/<call_id>.json`, amends the system prompt with the static rules + any prior-call findings).

  **Composition primitives**: `with_system_prompt(str | SystemMessage)` / `with_system_prompt_template(name, **vars)` / `with_runtime_system_prompt_template(template)` (all three mutually exclusive for the BASE prompt ΓÇö last wins). The runtime variant re-renders the system prompt from agent state on every model call (via internal `_RuntimePromptMiddleware`); when paired with `with_state_schema(...)`, only input-eligible fields (skipping `OmitFromSchema(input=True)` and reserved fields like `messages` / `structured_response`) flow as Jinja vars. `with_sandbox()` ΓåÆ default backend + registers all three cache tools (`discover_cached_tool_outputs`, `get_tool_output_schema`, `write_cached_tool_output_to_backend`) + appends `tool_result_cache.jinja` prompt; `with_short_term_memory(path="/scratch/")`; `with_persistent_memory(path="/memories/", filename="AGENTS.md")`; `with_skills(paths, *, skills_root=None, filter_middleware=None)` (deep-only, one bundle per agent); `with_subagents(Sequence[SubAgent | CompiledSubAgent])` (deep-only); `with_tool(tool, *, is_cacheable=True, run_limit=_DEFAULT_TOOL_RUN_LIMIT (6), thread_limit=None, exit_behavior="continue")` ΓÇö per-tool policy declared inline; `run_limit` defaults to 6 (constant `_DEFAULT_TOOL_RUN_LIMIT`) and automatically creates a per-tool `ToolCallLimitMiddleware`; pass `run_limit=None` to opt out of the per-tool cap; `with_permission(FilesystemPermission)` (deep-only); `with_middleware(AgentMiddleware)`; `with_response_format`, `with_store(BaseStore)`, `with_context_schema(type)`, `with_state_schema(type)` (ReAct-only ΓÇö extends `langchain.agents.AgentState`; use `OmitFromSchema` annotations on extra fields to declare input-only vs output-only when adding the compiled agent directly to a parent graph), `with_checkpointer(Checkpointer)`.

  **Auto-trigger: structured-response ΓåÆ state unpacking.** When BOTH `with_state_schema(...)` AND `with_response_format(...)` have been called, `_assemble_middleware` automatically wires `_StructuredResponseToStateMiddleware` ΓÇö it reads `state.structured_response` (the Pydantic instance) at `aafter_agent` and emits per-field state updates by attribute name. This is the unambiguous signal that the agent is meant to be added directly to a parent graph; each Pydantic field maps to a same-named state field. Scales to N output fields. The `trading_decision/analysts/*_analyst.py` factories use this pattern.

  **Universal middleware order** (set by `_assemble_middleware`, outermost first): `ModelCallLimitMiddleware` (cond) ΓåÆ `ToolCallLimitMiddleware`s (cond, may repeat) ΓåÆ `ModelFallbackMiddleware` (cond) ΓåÆ `ModelRetryMiddleware` (universal, transient-only filter) ΓåÆ `ContextEditingMiddleware` (cond) ΓåÆ `SummarizationMiddleware` (cond) ΓåÆ `ToolKnowledgeMiddleware` (universal) ΓåÆ `ToolResultCacheMiddleware` (universal) ΓåÆ `ToolRetryMiddleware` (universal, 5xx/network filter) ΓåÆ ReAct-only `FilesystemMiddleware` / `MemoryMiddleware` ΓåÆ `SubagentRefinementMiddleware` or `SubagentRefinementParentMiddleware` (cond) ΓåÆ optional skill-filter middleware ΓåÆ caller-supplied via `with_middleware`. Backend routes for `StateBackend()` / `StoreBackend(namespace=...)` / `FilesystemBackend(...)` are instantiated eagerly on capability registration; only the sandbox default (`get_backend`) runs as a `BackendFactory` closure.

- **`middlewares/`** ΓÇö Middleware package. Each subpackage is a focused concern:
  - `store_access/` ΓÇö `StoreAccessMiddleware` plus `AccessControlledStore` and 5 generic store CRUD `@tool` functions (`store_get` / `store_put` / `store_delete` / `store_search` / `store_list_namespaces`). `StoreConfiguration.store_allowed_namespaces: list[str] | None` restricts which namespace prefixes an agent can access (`None` = unrestricted).
  - `tool_result_cache/middleware.py` ΓÇö `ToolResultCacheMiddleware`: caches successful tool results in a shared `InMemoryStore` (via `ToolRuntime.store`) for cross-agent deduplication. **Strict-content invariant**: never mutates `ToolMessage.content` ΓÇö tools that return JSON keep returning JSON. Cache provenance lives in `additional_kwargs["cache"]`. Caches both `str` and `list` content (the two valid `ToolMessage.content` types per LangChain) ΓÇö this means Firecrawl tools (which return `list`) are cached and deduplicated. `_should_skip_cache(result, non_cacheable_patterns)` helper centralises skip logic: skips on `status='error'`, `is_error_content()` match (for `str` content), or any pattern in `non_cacheable_patterns` (for `str` content); `list` content with no error status is always cached. Size-based offloading of oversized payloads is **not** this middleware's job ΓÇö that's `deepagents.middleware.filesystem.FilesystemMiddleware._aintercept_large_tool_result` (default 20K-token threshold). `cacheable_tools` parameter restricts which tools are cached (None = all). `self.tools = []` ΓÇö cache tools are NOT registered here; they require sandbox and are registered by `with_sandbox()`. Also exports `get_args_hash()` and `is_error_content()` helpers.
  - `tool_result_cache/config.py` ΓÇö `ToolResultCacheConfiguration(BaseConfiguration)`. `tool_schema_packages: list[str]` (default `["muffin_agent.tools", "muffin_agent.middlewares.store_access"]`) ΓÇö decouples tool schema package scanning. `non_cacheable_patterns: list[str]` (default `["tool result too large"]`, env `TOOL_RESULT_CACHE_NON_CACHEABLE_PATTERNS`) ΓÇö case-insensitive substrings matched against string content; results containing any pattern are excluded from cache. Default excludes `FilesystemMiddleware` offload messages which embed ephemeral `tool_call_id`-keyed paths that become stale on cache hit.
  - `tool_result_cache/tools.py` ΓÇö `discover_cached_tool_outputs`, `get_tool_output_schema`, `write_cached_tool_output_to_backend`. **Registered by `with_sandbox()` only** ΓÇö not by `ToolResultCacheMiddleware.tools` (which is empty). Non-sandbox agents benefit from automatic transparent cache hits in `awrap_tool_call` without needing the explicit cache tools in their prompt.
  - `tool_knowledge/` (replaces the older `tool_error_handler/`) ΓÇö adaptive learning from tool failures. Split into focused modules:
    - `errors.py` ΓÇö `is_permanent_error(msg)` and `duplicate_key(tool_call)` (pure logic).
    - `lessons.py` ΓÇö `Lesson` dataclass + `LessonCatalog` wrapping a `BaseStore`; CRUD under namespace `("tool_lessons", tool_name)` keyed by `error_class_hash`.
    - `summariser.py` ΓÇö `summarise_tool_failure(...)` (LLM-distilled lesson) with `fallback_lesson(tool, error)` (deterministic) for the no-summariser path.
    - `prompt.py` ΓÇö `render_lessons_block(lessons)` and `append_block(existing, block)` for the system-prompt addendum.
    - `middleware.py` ΓÇö `ToolKnowledgeMiddleware`: catches tool exceptions and errored `ToolMessage`s, blocks duplicate permanent failures (carried via `failed_tool_calls` state), records a lesson per unique `(tool, error_class)` pair (LLM-summarised when `with_tool_knowledge(summariser)` was called, deterministic fallback otherwise). **Extended permanent-error detection**: beyond `status='error'` and raised exceptions, also catches `ToolMessage`s with `status=None` whose string content matches `is_permanent_error()` (e.g. "OpenAI API key is missingΓÇª") ΓÇö wraps the error as `"permanently unavailable in this environment: <content>"` so the recorded lesson is agent-facing (agent cannot configure API keys; lesson tells it not to retry). **Loop-pattern detection** (`loop_warning_threshold=3`): `awrap_model_call` counts tool calls per name across message history via `_count_tool_calls_in_history`; when any tool reaches the threshold, `_record_loop_lesson` is called with actual tool output samples (via `_get_recent_tool_outputs`) so the summariser can reason about why the tool isn't making progress (e.g. auth-gated content). Loop lessons use the same `LessonCatalog` + `summarise_tool_failure` pipeline as error lessons and flow into the same `## Lessons learned` block. `awrap_model_call` records loop lessons BEFORE reading the catalog so just-persisted lessons appear in the same call's block.
  - `subagent_refinement/` ΓÇö generic conversational refinement protocol so subagents can be re-invoked to fill gaps without restarting:
    - `schema.py` ΓÇö `CollectionFindings`, `Gap`, `GapReason` (StrEnum), `ToolCallSummary`. Generic `obtained: dict[str, Any]` so each subagent picks its own field names.
    - `storage.py` ΓÇö `call_id_path`, `extract_prior_call_id` (regex with word-boundary lookahead), `read_findings`, `write_findings`, `latest_human_text` helpers.
    - `prompts.py` ΓÇö thin facade over `render_template` that loads the role-specific instructions from the dedicated template directory (see "Prompt templates" note below) and renders the per-call prior-findings block.
    - `middleware.py` ΓÇö two classes:
      - `SubagentRefinementMiddleware` (child) ΓÇö `abefore_agent` discovers `prior_call_id=<id>` markers and stashes the prior findings on `prior_findings` state, `awrap_model_call` amends the system prompt with the static rules + (when present) the per-call prior block, `aafter_agent` writes this run's `CollectionFindings` to `/scratch/subagent_runs/<call_id>.json`. Auto-assigns `call_id` if missing.
      - `SubagentRefinementParentMiddleware` (parent) ΓÇö only `awrap_model_call`; prepends the parent-side rules so the orchestrator knows how to read `gaps` and re-issue refinement calls with `prior_call_id=<id>`. No state, no tools.
  - `skill_suggestion/middleware.py` ΓÇö `SkillFilterMiddleware(AgentMiddleware)`: schema-driven skill filtering middleware. Works alongside the default `SkillsMiddleware` (added via `skills=` on `create_deep_agent`). No tools registered. Parameterised via `__class_getitem__` with an `AgentState` subclass whose extra fields (beyond `AgentState`) become category keys: `SkillFilterMiddleware[TickerClassification](context_header=..., context_intro=..., context_outro=...)`. The three `context_*` constructor kwargs control the domain-specific text injected into the system prompt (defaults are domain-agnostic ΓÇö "Classification" / "The current context has been classified as follows:" / standard outro). Classification is read from flat state keys (not a nested dict), so multiple agents can compose without key conflicts. `_filter_skills()` keeps skills whose ALL metadata category values match the classification; universal skills (no metadata) always match. `abefore_agent` filters `skills_metadata`, `awrap_model_call` injects classification context into the system prompt. Used by `criteria_definition.py` (ticker classification) and `agents/research/` (mode + task_type filtering).

**Universal model-retry layer.** `MuffinAgentBuilder._assemble_middleware` wires LangChain's upstream `ModelRetryMiddleware` to retry transient/mid-stream LLM provider errors that escape the SDK's connect-time `max_retries` (e.g. `openai.APIError("Provider returned error")` raised from inside the SSE stream after a 200 OK). Configured with `max_retries=3`, exponential backoff (`backoff_factor=2.0`, `initial_delay=1.0`, `max_delay=30.0`, `jitter=True`), and a `retry_on=` callable (`_should_retry_llm_call` in `agent_builder.py`) that explicitly excludes permanent errors (`AuthenticationError`, `PermissionDeniedError`, `BadRequestError` for both `openai` and `anthropic`) before matching transient ones. `on_failure="error"` re-raises after exhaustion so an outer `ModelFallbackMiddleware` can switch models. Knobs are hardcoded ΓÇö tune by appending a custom `ModelRetryMiddleware` via `with_middleware(...)` if needed.

**Universal tool-retry layer.** Same file wires `ToolRetryMiddleware` (`max_retries=1`, `on_failure="continue"`, exponential backoff) with a `_should_retry_tool_call` filter that targets `ToolException` messages containing 5xx / gateway / connection / timeout substrings (plus bare `TimeoutError` / `ConnectionError`). 4xx, validation, and missing-credential errors propagate so the LLM (and `ToolKnowledgeMiddleware`) can adapt instead of burning retry budget. Sits below `ToolResultCacheMiddleware` so cache hits short-circuit and don't consume retry budget.

**Prompt templates owned by middlewares.** Capability prompt fragments live under `prompts/middlewares/`. Most are static partials auto-appended by `MuffinAgentBuilder` via `_PromptConfig.add_partial` (e.g. `tool_result_cache.jinja`, `short_term_memory.jinja`, etc.). The `subagent_refinement` capability owns its prompt content fully ΓÇö templates live in `prompts/middlewares/subagent_refinement/{child_instructions,parent_instructions,prior_findings_block}.jinja`, loaded by `subagent_refinement/prompts.py` via `render_template` and amended onto `system_message` from inside `awrap_model_call`. The agent builder is **not** responsible for prompt construction for capabilities whose middleware owns its own prompt.

- **`agents/data_collection/`** ΓÇö Data collection agents using MCP tools with the ReAct pattern. Each agent is a single `.py` file containing:
  - `MCP_TOOLS` ΓÇö list of allowed MCP tool name strings
  - `create_*_agent(config: RunnableConfig)` ΓÇö async; calls shared `get_tools()` for MCP tools, then composes the agent via `MuffinAgentBuilder(...).with_system_prompt_template(...).with_short_term_memory().with_tool(...).build_react_agent()`. Tools are cacheable by default; pass `is_cacheable=False` to opt out (e.g. `execute_python`).

  Shared utilities live in `utils.py`:
  - `get_tools(config: RunnableConfig, allowed_tools, custom_tools=None)` ΓÇö calls `McpConfiguration.from_runnable_config(config)` to get all MCP connections (OpenBB + Firecrawl), loads all tools via `MultiServerMCPClient`, then filters by `allowed_tools`. Skips MCP entirely when `allowed_tools=[]`.

  Currently implemented: `equity_fundamentals.py` (25 OpenBB tools), `equity_price.py` (5 OpenBB tools + `execute_python`), `web_search.py` (6 Firecrawl MCP tools; SearxNG accessed via Firecrawl's `SEARXNG_ENDPOINT`)

- **`agents/data_validation.py`** ΓÇö Pure reasoning agent (no MCP tools) that validates collected data against a criterion. Built with `create_agent(model, system_prompt=...)` and no tools ΓÇö the ReAct loop resolves to a direct LLM response. Checks sufficiency, relevance, temporal validity, and consistency. Prompt: `data_validation.jinja`. Used as a `CompiledSubAgent` in both stock evaluation and criterion evaluation agents.

- **`agents/subagents.py`** ΓÇö Shared subagent builders. `build_analysis_subagents(config)` creates the standard 15 subagents (14 data collection + 1 validation) used by `stock_evaluation.py` and `criterion_evaluation.py`. `build_validation_subagent(config)` creates just the data-validation subagent, used by all 4 investment stage agents.

- **`agents/investment/schemas.py`** ΓÇö Shared Pydantic models used across investment stage agents (`DataSource`).

- **`agents/investment/utils.py`** ΓÇö Shared `run_deep_agent_node()` utility that encapsulates the pattern common to all investment node functions: `ModelConfiguration` extraction ΓåÆ agent creation (with optional `store` param) ΓåÆ state context ΓåÆ ainvoke ΓåÆ structured output extraction ΓåÆ error fallback with exception handling. Accepts `store: BaseStore | None` and passes it to agent factories as `store=store` kwarg.

- **`agents/criteria_analysis/`** ΓÇö Criteria-driven analysis orchestrator graph. Top-level pipeline that:
  1. **Stage 1 ΓÇö `ticker_classification_node`**: lightweight deep agent (3 data subagents: etf-index, equity-fundamentals, economy-macro + data-validation) producing `TickerClassificationOutput` (sector / sub_sector / market / stock_type).   2. **Stage 2 ΓÇö `criteria_definition_node`** (parallel with Stage 3): thin wrapper around `create_criteria_definition_agent` reading the flat classification keys to filter `/skills/valuation/`.
  3. **Stage 3 ΓÇö `valuation_methodology_node`** (parallel with Stage 2): deep agent with `web-search` + `discovery-screening` subagents producing `ValuationMethodologyOutput` (methodology summary + 2ΓÇô5 ticker-specific additional criteria).   4. **Stage 4a ΓÇö `merge_criteria_node`**: pure-Python deterministic dedup. Lowercases + strips punctuation ΓåÆ canonical name; skill version wins on tie; re-normalises weights to sum to 1.0; tags each survivor with `source: "skill" | "web"`.

  6. **Stage 5 ΓÇö `synthesis_node`**: reasoning-only deep agent (no subagents, no tools) producing `CriteriaAnalysisSynthesis` ΓÇö composite score (weight-weighted), buy/sell/hold signal, weighted breakdown, key positives/negatives, divergences, confidence (floor of classification + weighted criterion confidences), thesis paragraph.

- **`multi_agent/`** ΓÇö Generic, reusable multi-agent **conference** framework. Use when several agents with different system prompts need to talk to each other (debate, collaborate, reach consensus). Public entry: `build_conference_graph(*, participants, moderator, terminator, judge=None, state_schema=ConferenceState, messages_field="messages", next_speaker_field="next_speaker", agent_cursors_field="agent_cursors", verdict_field="verdict", checkpointer=None) -> CompiledStateGraph`. Generates a `StateGraph` with one named node per participant + a pure-Python `dispatch` node; conditional edges route by `state[next_speaker_field]`.

  **Four pluggable Protocols** (all `@runtime_checkable`):
  - `Participant` ΓÇö `name: str` + `async speak(state, config) -> str`. Three built-in implementations:
    - `LLMParticipant(name, system_prompt_template, llm_role="reasoner", user_prompt="Take your turn now.")` (Option ╬▒ ΓÇö conversation rendered into the system prompt as text; matches the risk_debate migration's zero-prompt-churn path).
    - `LLMMessageParticipant(...)` (Option ╬▓ ΓÇö conversation forwarded as a `BaseMessage` thread; system prompt holds only the role description).
    - `AgentParticipant(name, agent, user_prompt="Take your turn now.")` ΓÇö wraps a compiled muffin agent (ReAct or deep). Added as a parent-graph node via a thin framework-generated `prep ΓåÆ agent ΓåÆ extract` subgraph; per-agent state persists across turns via the agent's own LangGraph per-thread checkpointer. **Caller must build the agent with `MuffinAgentBuilder(...).with_checkpointer(True)`** (the langgraph `Checkpointer = None | bool | BaseCheckpointSaver` sentinel ΓÇö `True` enables per-thread persistence as documented in [LangGraph subgraph docs](https://docs.langchain.com/oss/python/langgraph/use-subgraphs)). The parent graph or conference must be compiled with a real checkpointer instance for the per-thread mechanism to engage. `AgentParticipant` does NOT implement the `Participant` Protocol's `speak()` ΓÇö it's wired via `build_conference_graph`'s subgraph-wrap path instead.
  - `Moderator` ΓÇö `next_speaker(state) -> str`. Built-ins: `RoundRobinModerator(speaker_order)`, `AlternatingModerator(speaker_a, speaker_b)` (two-speaker lead-count alternation; matches the legacy `_route_investment_debate` exactly so a future Bull/Bear migration is mechanical). Counts speakers by reading `msg.name` on each entry in shared `messages`.
  - `Terminator` ΓÇö `should_stop(state) -> (stop: bool, reason: str | None)`. Built-in: `MaxRoundsTerminator(max_rounds, num_participants)` (1 round = each participant has spoken once).
  - `Judge` ΓÇö optional post-conference synthesiser. Built-in: `StructuredOutputJudge(name, system_prompt_template, output_schema, llm_role="reasoner")` (one LLM call with `with_structured_output(schema)` via `ModelConfiguration.get_chat_model_for_role(config, role, schema=schema)`; returns `result.model_dump()`).

  **Shared state ΓÇö messages-only.** Single source of truth: `messages: Annotated[list[BaseMessage], add_messages]`. Each turn appends one `AIMessage(content, name=<speaker>, id=<uuid>)`. The `name` attribute is set explicitly by the framework wrapper so cursor tracking + chronological rendering work portably. Per-speaker views: `render_messages_chronological(messages)` (joined `"<name>: <content>"` lines) and `last_opposing_message(messages, speaker)`. Per-agent cursor: `agent_cursors: dict[str, str]` maps speaker name ΓåÆ last seen `BaseMessage.id`. When the conference is embedded as a subgraph in a parent graph with renamed fields (e.g. `risk_debate_messages` / `risk_debate_agent_cursors`), pass `messages_field=...` / `agent_cursors_field=...` to the builder and ensure the parent state schema declares them.

  **Topology**: `START ΓåÆ dispatch ΓåÆ [participant_1, ..., participant_N] ΓåÆ dispatch ΓåÆ ... ΓåÆ [judge | END]`. `dispatch` calls `terminator.should_stop` first; on stop it clears `next_speaker` and routes to the judge (if configured) or END. Otherwise `moderator.next_speaker` sets the routing target. LLMParticipants run via a Python adapter node that calls `speak()` and appends one tagged `AIMessage` to `messages_field`. AgentParticipants run via a `prep ΓåÆ agent ΓåÆ extract` subgraph where the prep step slices messages by cursor (only new since this agent's last turn) and converts other-speaker `AIMessage`s to `HumanMessage(f"[{speaker}]: {content}", name=speaker)` so the agent's chat-completions endpoint sees the expected alternating user/assistant pattern; the agent (compiled with `checkpointer=True`) runs its full ReAct loop and persists its state across turns via the per-thread namespace; the extract step tags the agent's final `AIMessage` with `name=participant.name` and updates the cursor.

  **No leak invariant**: the agent's internal tool calls / intermediate AIMessages stay in its own per-thread checkpoint. The wrapper's `extract` node emits `RemoveMessage(id=...)` entries to clear them from the subgraph state before propagation, so parent conference state only ever sees one tagged AIMessage per turn.

  **Current production consumer**: `agents/trading_decision/graph.py:_build_risk_debate_subgraph(max_rounds)` builds the 3-way Aggressive/Conservative/Neutral risk-debate subgraph as a single `add_node("risk_debate", ...)` in the parent. Bull/Bear (investment debate) is intentionally untouched today (see roadmap follow-up).

  **Out-of-scope (deferred)**: LLM-driven `Moderator` (Manager decides next speaker by reading the transcript), consensus-detecting `Terminator` (early stop when participants agree) ΓÇö extension slots for future use cases. Files: `multi_agent/{conference,participants,moderators,terminators,judges,state,_formatters,__init__}.py` + `prompts/multi_agent/_transcript.jinja` (optional partial). Tests: `tests/multi_agent/`.

- **`agents/research/`** ΓÇö General-purpose deep research subgraph (domain-agnostic, Perplexity-style). Linear LangGraph pipeline: `classifier ΓåÆ researcher ΓåÆ rerank ΓåÆ writer`. Inspired by Vane (Perplexica) ΓÇö single-classifier-with-flags shape, embedding-rerank step (cosine ΓëÑ 0.5, top-K 20), inline `[N]` citation discipline. Nodes:
  1. **`classifier_node`** ΓÇö plain ReAct agent (no tools, collector role) with `response_format=AutoStrategy(ResearchClassification)`.   2. **`researcher_node`** ΓÇö deep agent built via `MuffinAgentBuilder`.   3. **`rerank_node`** ΓÇö pure Python (no LLM).   4. **`writer_node`** ΓÇö plain ReAct agent (no tools, orchestrator role) with `response_format=AutoStrategy(ResearchOutput)`. Pre-pass computes `allowed_actions` per ticker (max `buy`/`sell`/`short`/`cover` shares within cash/margin/position-limit constraints); pre-fills `hold` for any ticker with no legal non-hold action (skips LLM entirely when all tickers are pre-filled). LLM call (with `PortfolioReconciliationOutput` schema) picks final orders + writes `portfolio_notes`.

- **`agents/portfolio_decision.py`** ΓÇö Multi-ticker paper-trading decision graph. `Send ├ù N tickers` ΓåÆ per-ticker subgraph (council ΓåÆ `ticker_decision_node`) ΓåÆ barrier ΓåÆ `position_sizing_node` ΓåÆ `portfolio_reconciler_node` ΓåÆ `execute_orders` (applies orders via `portfolio.executor.apply_orders`) ΓåÆ END. Returns updated `portfolio` snapshot + `executed_trades` + `portfolio_value`. CLI: `muffin trade AAPL,MSFT --portfolio my-port [--apply]`. Portfolio JSONs under `~/.muffin/portfolios/<name>.json`. Full guide: [docs/paper-trading.md](docs/paper-trading.md).

- **`backtesting/`** ΓÇö Walk-forward backtester. `BacktestEngine(start, end, tickers, mode, rebalance_freq="ME", benchmark="SPY", risk_free_rate=0.0434)` iterates `pd.date_range(...)`, invokes either `portfolio_decision_graph` (mode=`full`) or `council_graph` only (mode=`signals`) at each date. Caller supplies `prices_provider` (async `(date) -> dict[ticker, price]`) + optional `benchmark_provider`. Returns `BacktestResults(history, final_portfolio, final_value, metrics)` with `to_dataframe()` view. `metrics.py`: `compute_sharpe`, `compute_sortino`, `compute_max_drawdown`, `compute_total_return`, `compute_returns_from_equity`, `compute_benchmark_comparison` ΓÇö pure-Python (statistics + math, no scipy). `synthetic_price_provider(base_prices, daily_drift)` is the test fixture. `muffin backtest` CLI is a configuration stub today; MCP-backed default `prices_provider` is a roadmap item. Full guide: [docs/backtester.md](docs/backtester.md).

- **`utils/outcomes.py`** ΓÇö Canonical re-export of `OutcomesFetcher` Protocol + `fetch_decision_outcome` default implementation (originally in `agents/trading_decision/tools.py`; kept there for backward compatibility). This is the discoverable entry point for any reflection-style consumer (today: `trading_decision` reflection bookends; future: backtester reflection mode).

## Memory

Filesystem routes are composed opt-in through `MuffinAgentBuilder`. Every capability the builder opts into comes with (a) a `CompositeBackend` route, (b) the matching prompt partial teaching the LLM how to use it, and (c) any required middleware. The four available routes:

- **`/memories/`** (long-term, per-user) ΓÇö enabled via `.with_persistent_memory()`. Backed by `StoreBackend` namespaced to `("memories", user_id)`. Deep agents get the stock `MemoryMiddleware` implicitly via `create_deep_agent(memory=[...])`; ReAct agents get `MemoryMiddleware` wired explicitly in `_assemble_middleware`. Both auto-load `AGENTS.md` into the system prompt. `_memories_namespace` resolves the namespace in three steps: `configurable.user_id` ΓåÆ `MemoryConfiguration.memory_debug_user_id` (env `MEMORY_DEBUG_USER_ID` or `configurable.memory_debug_user_id`) as a debug-only fallback ΓåÆ raise `MemoryUnavailableError`. CLI: pass `--user <id>`. Server: set `configurable.user_id` per request (via LangGraph auth hook once multi-user is actually required). Dev stack: set `MEMORY_DEBUG_USER_ID` in `.env` or on the `langgraph-api` service ΓÇö do NOT set in multi-user deployments.
- **`/scratch/`** (short-term, thread-scoped) ΓÇö enabled via `.with_short_term_memory()`. Backed by `StateBackend`, checkpointed with graph state. Survives `execute_python` sandbox container recycling; cleared when the thread is deleted.
- **`/skills/`** (read-only) ΓÇö enabled via `.with_skills(paths, filter_middleware=...)` (deep-agent only). Backed by `FilesystemBackend(virtual_mode=True)` rooted at the shipped `skills/` directory. Currently used by `criteria_definition.py` (`/skills/valuation/`) and `agents/research/` (`/skills/research/` ΓÇö modes + task_types). Research evidence cache lives under `/scratch/research/<thread_id>/` (created on demand by the researcher).
- **Default (no prefix)** ΓÇö enabled via `.with_sandbox()`. Routes to `OpenSandboxBackend` ΓÇö where `execute_python` runs and where auto-offloaded tool outputs (>20k tokens) land. Ephemeral.

User isolation: the `user_id` key flows from `--user` (CLI) ΓåÆ `configurable={"user_id": ...}` ΓåÆ `_memories_namespace(rt)`. Each user gets an isolated namespace. Local dev against clients that don't populate `user_id` (e.g. `agent-chat-ui`) set `MEMORY_DEBUG_USER_ID` so the fallback pins them to a shared namespace; without it `_memories_namespace` raises `MemoryUnavailableError`. Multi-user deployments must populate `configurable.user_id` per request to get isolation and must NOT set `MEMORY_DEBUG_USER_ID` (it collapses all anonymous traffic onto a single shared namespace).

Data-collection ReAct sub-agents opt into `/scratch/` via `.with_short_term_memory()` and get `FilesystemMiddleware` wired automatically; they do **not** call `.with_persistent_memory()`.

## Conventions

- **Ruff** with Google-style docstrings (`D401` imperative mood enforced). `D` and `UP` rules are relaxed in `tests/`.
- **Pydantic** models for all configuration and structured outputs.
- Python 3.11+ required.
- Environment variables for secrets (API keys, LangFuse credentials). See `.env` file.
- MCP (Model Context Protocol) for external data access (OpenBB).
- Use `create_agent` from `langchain.agents` for ReAct agents ΓÇö do NOT use `create_react_agent` from `langgraph.prebuilt` (deprecated).
- KISS: no empty files or premature abstractions. Extract shared utilities only when patterns emerge across multiple agents.

## Collaboration Preferences

These rules govern how Claude approaches planning, implementation, and communication in this project.

1. **Deep planning first** ΓÇö Always do deep planning and trade-off evaluation before writing any code. Explore the solution space thoroughly before committing to an approach.

2. **Prefer out-of-the-box solutions** ΓÇö Before implementing custom logic, research available library features by reading internet documentation and/or library source code. Consider alternative options even if they are not an exact match to the ask. Surface interesting options proactively.

3. **Propose options, don't decide** ΓÇö When facing a design decision or when multiple approaches exist, present the options and ask for a decision rather than picking one unilaterally. Ask questions before writing substantial code if no existing library/utility has been found ΓÇö the user may be able to provide documentation pointers.

4. **Explicit approval before implementation** ΓÇö Always ask for explicit approval before starting implementation. Never exit plan mode unless the user explicitly says to exit or switch mode.

5. **Keep documentation up to date** ΓÇö After every implementation, update README.md, docs/, roadmap.md, and any other relevant docs as applicable. Add VSCode launch configurations where reasonable. Always include documentation updates as the last step of implementation plans. When trade-offs or tech debt are accepted, document the limitations and add action items to roadmap.

6. **Memorize lessons in CLAUDE.md** ΓÇö If the user shares information that will be useful in future sessions (e.g. future roadmap tasks, corrections, disagreements, repeating feedback patterns, new constraints), record it in CLAUDE.md. When in plan mode, include the CLAUDE.md memory update as an explicit plan step.
